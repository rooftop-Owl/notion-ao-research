이 규칙들은 Notion 워크스페이스를 수년간 만들고 부수고 다시 만든 경험에서 나왔습니다. 오래가는 시스템과 이쁜 쓰레기를 가르는 기준이기도 합니다.

Notion 실패의 대부분은 기술적 실패가 아니라 설계 실패입니다. 빈 페이지를 열고, 꾸미고, 하루 저녁 뿌듯해하다가 2주 만에 버립니다. 왜? 시스템이 너무 많은 입력을 요구하고, 시각적 노이즈 뒤에 중요한 상태를 숨기고, 단일 진실 공급원(Single Source of Truth)이 없기 때문입니다. 좋은 워크스페이스는 스크린샷에서 멋져 보이는 게 아닙니다. 피곤하고, 바쁘고, 짜증 날 때도 쓸 수 있는 워크스페이스가 좋은 워크스페이스입니다.

이 장은 의도적으로 독단적입니다. 규칙이 엄격하게 느껴지더라도, 깰 만한 근거가 충분히 쌓이기 전까지는 그대로 지키십시오. 시스템 설계에서는 일관성이 창의성을 이깁니다.

---

## 세 가지 원칙

### 1) 심플하게 (Keep It Simple)

예쁜 것과 심플한 것은 다릅니다. 과도한 꾸밈은 **이쁜 쓰레기**입니다.

Notion은 시각적 장난감을 너무 많이 줍니다: 아이콘, 커버, 콜아웃, 구분선, 이모지 라벨, 미적 대시보드. 이것들 자체가 나쁜 건 아닙니다. 입력과 조회를 방해할 때 나빠집니다. 시스템은 데이터를 캡처하고 꺼내 보기 위해 존재합니다. 꾸밈이 이 두 단계를 느리게 한다면, 그건 디자인이 아니라 마찰입니다.

> [!IMPORTANT]
> **규칙:** 데이터 입력이나 조회에 기여하지 않는 디자인 요소는 제거하십시오.

실전 해석:
- 한 문장으로 설명할 수 없는 요소("입력을 빠르게 해준다" 또는 "기록을 빨리 찾게 해준다")는 삭제하십시오.
- 색상 의미를 설명하는 긴 범례가 필요하다면, 구조가 약한 겁니다.
- 속성 정의보다 블록 정렬에 시간을 더 쓰고 있다면, 시스템이 아니라 연극을 만들고 있는 겁니다.

심플함은 미학적 미니멀리즘이 아닙니다. 운영적 명확성입니다. 밤 11시 30분에 워크스페이스를 열어도 어디에 입력하고, 뭘 확인하고, 다음에 뭘 해야 하는지 바로 알 수 있어야 합니다.

### 2) 직관적으로 (Make It Intuitive)

직관은 마법이 아닙니다. 일관된 규칙의 결과입니다.

사람들이 워크스페이스를 "직관적이지 않다"고 말하는 건, 시스템이 자기 언어를 계속 바꿀 때입니다. 한 테이블은 이모지 상태를 쓰고, 다른 테이블은 텍스트 상태를 쓰고, 세 번째는 라벨 없는 아이콘을 씁니다. 한 페이지에서 노란색이 긴급을 뜻하고, 다른 페이지에서는 입력 영역을 뜻합니다. 사용자가 실패하는 건 부주의해서가 아니라 시스템이 일관성이 없기 때문입니다.

> [!IMPORTANT]
> **규칙:** 자체 가이드라인을 정하고 지키십시오. 처음 보는 사람이 30초 안에 데이터 입력 위치를 찾을 수 있습니까?

핵심 규칙 예시 (사용하거나 변형하되, 무작위로 섞지 마십시오):
- **노란색 = 입력 구역** (Yellow = input area)
- **테이블 아이콘 = 마스터 테이블** (Table icon = master table)
- **아이콘 없음 = 일반 데이터 페이지** (No icon = regular data page)

> [!TIP]
> **30초 테스트:** 페이지를 모르는 사람에게 보여주십시오. 30초 안에 데이터 입력 위치를 찾지 못하면, 설계가 실패한 겁니다. 변명하지 말고 다시 설계하십시오.

### 3) 일이 되지 않게 (Don't Make It a Chore)

Notion의 가장 큰 위험은 복잡성이 아닙니다. 피로감입니다.

사무 노동처럼 느껴지는 워크플로우는 죽습니다. 새 행마다 같은 날짜, 카테고리, 관계, 상태를 반복 입력해야 한다면, 결국 데이터 입력을 멈추게 됩니다. 입력이 멈추면 모든 대시보드는 낡은 정보가 되고, 모든 수식은 허구가 됩니다.

> [!IMPORTANT]
> **규칙:** 같은 값을 3번 이상 입력하고 있다면, 자동화하십시오.

자동화 기본선:
- 레코드 생성 시 날짜 자동 입력.
- 맥락이 명확할 때 기본 유형 자동 분류.
- 공통 관계 자동 연결 (또는 제한된 빠른 선택 경로 제공).
- 파생 필드는 수식/규칙으로 처리. 수동 복사-붙여넣기 금지.

"일이 되지 않게"는 "화려하게 만들라"는 뜻이 아닙니다. 반복 키 입력을 없애라는 뜻입니다. 반복되는 수동 단계는 모두 빚입니다. 빚은 일찍 갚으십시오.

> [!NOTE]
> **핵심 포인트:** 세 가지 원칙은 우선순위 스택입니다. 충돌할 때: 심플하게 > 직관적으로 > 일이 되지 않게. 약간 수동적이지만 심플한 시스템이, 아무도 이해 못하는 복잡한 자동화 시스템보다 낫습니다.

## 아이콘 규칙

기본 정책: **데이터 페이지에는 아이콘 없음** (No icons on data pages).

이 기본값이 중요한 이유:
1. **아이콘은 콘텐츠 상태를 숨깁니다.** Notion에서 콘텐츠 유무의 시각적 단서는 유용합니다. 임의의 아이콘이 그 신호를 평탄화합니다.
2. **아이콘 선택이 노동이 됩니다.** 수백 가지 선택지가 운영적 이득 없이 결정 피로만 만듭니다.
3. **시각적 노이즈는 규모에서 악화됩니다.** 아이콘 몇 개는 무해하지만, 수백 개는 스캔 가능성을 파괴합니다.

> [!WARNING]
> "예뻐 보이니까" 모든 페이지에 아이콘을 넣으면 유지보수 부채를 만드는 겁니다. 200개 이상의 페이지가 생기면 아이콘 관리 자체가 일이 됩니다. 시작하기 전에 멈추십시오.

아이콘은 다음 네 가지 예외 중 **하나 이상**에 해당할 때만 사용하십시오:

| # | 예외 | 아이콘이 도움이 되는 이유 |
|---|------|--------------------------|
| 1 | **네비게이션 버튼** (Navigation buttons) | 클릭 대상으로 존재하는 페이지 — 아이콘이 인터랙티브하게 보이게 함 |
| 2 | **마스터 테이블** (Master tables) | Linked View와 빠르게 구분해야 하는 핵심 데이터베이스 |
| 3 | **카테고리/관계형 페이지** (Category/relation pages) | 롤업에서 시각적 그룹핑이 인식을 향상시키는 분류 노드 |
| 4 | **외부 공개형 페이지** (External-facing pages) | 마케팅/채용/쇼케이스 등 프레젠테이션이 최우선 목표인 경우 |

이 네 가지 기준에 해당하지 않는 페이지는 아이콘을 제거하십시오. 논쟁 없이. 이 단일 규칙만으로 대규모 워크스페이스의 명확성이 극적으로 향상됩니다.

## 네이밍 규칙

네이밍은 행동 설계입니다. 속성 이름은 필드가 무엇인지가 아니라, 사용자가 무엇을 해야 하는지를 알려줘야 합니다.

### 1) 마스터 테이블 이름은 영어로

마스터 테이블 이름은 영어로 유지하고 안정적으로 관리하십시오. 언어 선호의 문제가 아니라, 일관된 인덱싱과 페이지 간 가독성의 문제입니다:
- `Tasks`, `Projects`, `Meetings`, `References`, `Contacts`

> [!TIP]
> 테이블 이름을 매달 바꾸지 마십시오. 안정성이 참신함보다 가치 있습니다.

### 2) `입력완료 ✓` — 트리거 패턴

`입력완료 ✓` (Input Complete ✓)처럼 명확한 트리거 필드명을 사용하여 라이프사이클 의도를 전달하십시오. 체크마크는 장식이 아니라 워크플로우 계약입니다.

| ❌ 나쁜 이름 | ✅ 좋은 이름 |
|-------------|--------------|
| `done2` | `입력완료 ✓` |
| `flag` | `Input Complete ✓` |
| `ok?` | `Ready for Processing ✓` |

### 3) 노란색 배경 규칙

노란색은 "여기서 입력한다"는 뜻입니다. 긴급이 아닙니다. 중요가 아닙니다. 경고가 아닙니다. **입력 전용.**

노란색이 한 페이지에서 한 가지를 뜻하고 다른 페이지에서 다른 것을 뜻하면, 사용자는 색상 단서에 대한 신뢰를 잃습니다. 규칙은 지루할 정도로 일관적일 때만 작동합니다.

### 4) 라이프사이클 순서의 상태 옵션

상태 옵션은 프로세스 순서를 따라야 합니다. 감정적 순서나 무작위 선호가 아닙니다.

```text
✅ 라이프사이클 순서:  Inbox → Input Complete → In Progress → Complete → Archived
❌ 무작위 순서:        Urgent, Later, Maybe, Final, Start
```

상태 필드는 전환 로직을 인코딩해야 합니다. 옵션이 순서를 암시하지 않는다면, 그건 라이프사이클 상태가 아니라 단순 라벨입니다.

## 안티패턴

> [!CAUTION]
> 이것들은 워크스페이스가 *활발해 보이면서* 조용히 사용을 죽이는 반복적인 실수입니다. 팀이 계속 "이거 어디에 쓰지?" 또는 "이거 왜 사라졌지?"라고 묻는다면, 아마 이 중 세 가지 이상을 동시에 저지르고 있을 겁니다.

| 안티패턴 | 왜 나쁜가 | 해결책 |
|---|---|---|
| **Notion을 메모 앱으로 사용** | 기능의 10% 정도만 활용; 정보가 재사용 가능한 속성 없는 비구조화 텍스트가 됨 | 처음부터 속성과 뷰가 있는 데이터베이스 사용 |
| **데이터 복제** | 단일 진실 공급원 없음; 복사본 간 충돌 발생, 어느 행이 맞는지 아무도 모름 | 마스터 테이블 하나를 유지하고 Linked View 사용 |
| **모든 것을 수동으로** | 2주 만에 번아웃; 반복 입력이 일관성과 동기를 파괴 | 기본값 자동 입력, 수식, 자동화 규칙 활용 |
| **모든 페이지에 아이콘** | 시각적 노이즈가 콘텐츠 상태를 숨기고 스캔 속도를 저하 | 위의 4가지 기준 아이콘 규칙 적용 |
| **입력 전용이 아닌 뷰에서 편집** | 필터 로직 깨짐; 레코드가 예상치 못하게 사라지거나 필수 필드를 우회 | 생성/편집을 Input Section으로 제한 |
| **복잡한 수식을 한 번에** | 디버깅 불가능; 하나의 오류가 다섯 가지 가정을 숨김 | 수식을 작은 조각으로 나누어 각 부분을 검증 |
| **과도한 꾸밈** | 데이터 품질이 아닌 레이아웃 연극에 시간 소비 | 구조 먼저, 꾸밈은 하지 않기 |

## 시스템을 살려두는 운영 규칙

1. **단일 진실 공급원이 먼저, 뷰는 그 다음.**
   "새 페이지" 요청은 보통 "새 데이터베이스"가 아니라 "새 뷰"를 의미합니다.

2. **입력 경로는 명확하고 빨라야 합니다.**
   속도를 위해 테이블 기반 입력을 선호하십시오. 키보드 친화적 워크플로우가 마우스 중심 워크플로우를 이깁니다.

3. **규칙이 취향을 이깁니다.**
   팀 수준의 일관성이 개인 미학보다 중요합니다.

4. **세 번째 반복 후에 자동화하십시오.**
   첫 번째 반복은 신호이고, 세 번째는 설계 버그입니다.

5. **분기별로 감사하십시오.**
   죽은 속성을 제거하고, 중복 상태를 병합하고, 30초 입력 발견 규칙을 다시 테스트하십시오.
