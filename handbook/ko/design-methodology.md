# 설계 방법론

> **이 장에서 배우는 것:** 오래가는 Notion 워크스페이스를 설계하는 완전한 아키텍처 — 첫 번째 데이터베이스부터 자동 데이터 흐름을 갖춘 멀티 테이블 시스템까지.
>
> - Master Table: 데이터 유형별 단일 진실 공급원
> - Linked Database View: 같은 데이터, 다양한 관점
> - Input Section: 도입 성패를 가르는 UX 패턴
> - 데이터 흐름: 입력 → 처리 → 완료 라이프사이클
> - 상태 라이프사이클 설계: 방향이 있는 여정으로서의 상태
> - 페이지 구조: 확장 가능한 조직을 위한 3가지 페이지 유형
> - 케이스 스터디: 9개 테이블 디지털 도서관 전체 구현

**함께 읽기:** [원칙과 안티패턴](./principles-and-antipatterns.md) — 이 방법론의 기반이 되는 3가지 규칙. · [수식과 자동화](./formulas-and-automation.md) — 데이터베이스를 자기 유지 시스템으로 만드는 방법.

데이터베이스를 하나라도 만들기 전에, Notion을 강력하게 만드는 아키텍처를 이해해야 합니다. 이건 메모 앱이 아닙니다 — 데이터 시스템을 구축하는 플랫폼입니다.

Notion을 빈 노트북처럼 다루면 빈 노트북을 얻게 됩니다: 흩어진 메모, 중복 페이지, 나중에 뭔가 찾으려 할 때마다 끊임없는 마찰. 하지만 Notion을 시스템 설계 문제로 접근하면 경험이 완전히 달라집니다. "페이지에 쓰는" 것을 멈추고 흐름을 구축하기 시작합니다: 데이터가 어디서 들어오고, 어떻게 처리되고, 어디서 소비되고, 언제 완료로 간주되는지.

> [!IMPORTANT]
> **좋은 Notion 워크스페이스는 일회성 셋업이 아니라 반복 사용을 위해 설계됩니다.** 예쁜 포맷팅은 하루 감동을 줄 수 있습니다. 마찰 없는 구조는 수년간 워크스페이스를 계속 쓰게 만듭니다.

워크스페이스를 집 부엌이라고 생각해 보십시오:

- 재료를 보관하는 믿을 수 있는 저장소 하나가 필요합니다.
- 준비, 조리, 서빙을 위한 각각 다른 작업대가 필요합니다.
- 매일 같은 양파를 다시 썰지 않도록 명확한 규칙이 필요합니다.

이 장이 구축하는 것이 바로 그것입니다: 하나의 진실 공급원, 목적에 맞게 구축된 다양한 뷰, 규율 있는 입력 흐름, 그리고 원시 입력에서 완성된 결과물까지의 명확한 라이프사이클.

## 마스터 테이블

모든 본격적인 Notion 워크스페이스는 **Master Table**에서 시작합니다: 하나의 도메인(책, 할 일, 실험, 연락처, 수업 등)에 대한 원본 레코드를 저장하는 단일 데이터베이스입니다.

이런 비유를 써 보겠습니다: 마스터 테이블은 **마스터 냉장고**입니다. 채소용 냉장고 하나, 우유용 냉장고 하나, 남은 음식용 냉장고 하나를 각각 다른 집에 두지 않습니다. 하나의 진짜 저장소를 두고, 요리할 때 뭘 꺼내서 어디에 놓을지 결정합니다.

Notion 용어로:

1. **하나의 테이블이 원본 레코드를 보관합니다.**
2. **나머지는 모두 같은 테이블의 뷰입니다.**
3. **같은 엔티티에 대해 병렬 데이터베이스를 만들지 않습니다.**

> [!TIP]
> **Master Table 규칙** — 첫 구축 시 엄격하게 적용하십시오:
> - **전체 페이지 형식**으로 생성 (데이터가 비좁지 않게 확장 가능)
> - **영어 이름** 사용 (템플릿과 팀 간 이식성)
> - **속성을 숨기지 않기** (이것이 감사 표면입니다)
> - **날짜순 정렬** (라이프사이클과 최신성이 보이도록)
> - **명확한 아이콘 표시** (진실 공급원으로 시각적으로 구분)

왜 이렇게 엄격해야 할까요? 숨겨진 구조가 숨겨진 오류를 만들기 때문입니다. Notion 피로의 대부분은 자기 워크스페이스에 대한 신뢰를 잃는 데서 옵니다 — "진짜 버전이 어디 있지?" "이 레코드는 왜 없지?" 마스터 테이블 패턴은 이런 혼란이 시작되기 전에 방지합니다.

> [!NOTE]
> **핵심 포인트:** 데이터 유형당 마스터 테이블 하나. 나머지는 모두 뷰. 같은 종류의 데이터에 대해 두 번째 데이터베이스를 만들려 한다면 — 멈추고 Linked View를 만드십시오.

## Linked Database View

마스터 테이블이 존재하면, 다음 규칙은 협상 불가입니다:

> [!IMPORTANT]
> **새 데이터를 만들지 마십시오 — 새 뷰를 만드십시오.**

Linked Database View는 별도로 관리하는 복사본이 아닙니다. 같은 데이터를 보는 서로 다른 창입니다. 원본 레코드를 한 번 변경하면 모든 뷰에 반영됩니다.

Notion은 6가지 뷰 형식을 제공합니다 — 묻고 있는 질문에 따라 선택하십시오:

| 뷰 | 최적 용도 |
|------|---------|
| **Table** | 고밀도 편집, 빠른 키보드 입력 |
| **Board** | 단계 기반 이동 (상태 칼럼) |
| **Gallery** | 시각적 스캔 (커버, 썸네일, 카드) |
| **List** | 컴팩트한 읽기, 선형 탐색 |
| **Calendar** | 날짜 기반 계획 및 회고 |
| **Timeline** | 기간 및 시간 축 겹침 |

같은 레코드, 다른 의미. 하나의 "Task" 레코드를 상상해 보십시오:

- **Table**에서는 빠르게 업데이트할 수 있는 행입니다.
- **Board**에서는 "In Progress" 아래에 나타납니다.
- **Calendar**에서는 마감일에 나타납니다.
- **Timeline**에서는 계획된 주간에 걸쳐 늘어납니다.

네 개의 할 일을 만든 게 아닙니다. 네 가지 관점을 만든 겁니다.

> [!WARNING]
> 초보자들이 어려움을 겪을 때, "오늘 할 일", "주간 할 일", "완료된 할 일" 같은 별도 데이터베이스를 만드는 경우가 많습니다. 처음에는 정리된 것 같지만, 곧 중복의 늪에 빠집니다. Linked View는 필터, 정렬 규칙, 그룹핑, 레이아웃 변경으로 이 문제를 영구적으로 해결합니다.

## Input Section

마스터 테이블이 뇌라면, **Input Section**은 손입니다.

> [!IMPORTANT]
> 이것은 배워야 할 가장 중요한 UX 패턴입니다. 대부분의 워크스페이스가 실패하는 이유는 리포팅이 약해서가 아니라, **데이터 입력이 짜증 나기 때문**입니다. 입력이 힘들어지면 일관성이 떨어집니다. 그러면 워크스페이스가 낡아집니다.

Input Section은 시각적으로 구분되는(보통 **노란색 배경**) 전용 입력 영역을 만들어 이 문제를 해결합니다. 새 레코드를 빠르게 캡처하는 곳입니다. 입력이 완료되면 레코드가 입력 영역에서 자동으로 사라지고 다음 단계로 이동합니다.

### 방법 1: 단순 필터 (입문용 최적)

다음 조건의 Linked Table View를 사용하십시오:
- `Complete = unchecked`
- 보이는 필드를 **3개 이하**로 유지 (Title, Date, Complete)

가장 빠른 도입 경로입니다. 입력 → 완료 체크 → 행이 입력 뷰에서 사라짐.

### 방법 2: OR 필수 로직 (더 풍부한 레코드용)

**필수 필드가 4개 이상**일 때, 핵심 필드 중 하나라도 비어 있으면 행이 보이는 OR 기반 조건을 사용하십시오:

```text
Title is empty  OR  Date is empty  OR  Priority is empty  OR  Complete is unchecked
```

모든 필수 값이 채워지고 완료가 체크되어야만 행이 입력에서 사라집니다.

### 방법 3: 기본값 자동 입력 (대규모 속도용)

예측 가능한 값을 규칙/기본 동작으로 미리 채우십시오 (오늘 날짜, 기본 카테고리, 기본 순위). 사용자는 여전히 의미 있는 필드를 소유하지만, 반복적인 보일러플레이트는 사라집니다.

> [!TIP]
> **첫 워크스페이스를 위한 실전 셋업:**
> - 입력 블록을 놓칠 수 없게 만드십시오 (노란색 규칙)
> - 페이지 상단 근처에 배치하십시오
> - 키보드 우선 속도를 위해 테이블 레이아웃 사용
> - 트리거 필드에 명확한 라벨: `입력완료 ✓`
>
> 목표: 미래의 자신이 건너뛸 핑계를 댈 수 없을 만큼 캡처를 심플하게 만드는 것.

## 데이터 흐름: 입력 → 처리 → 완료

마스터 테이블 + Linked View + Input Section을 결합하여 라이프사이클을 만드십시오. 데이터베이스 간에 데이터를 이동하는 것이 아닙니다 — 하나의 데이터베이스 안에서 레코드를 **상태**를 통해 이동시키고, 필터된 뷰가 각 단계를 나타냅니다.

```text
┌─────────────────────────────────────┐
│  입력 뷰                             │
│  필터: 입력완료 = false               │
│                                     │
│  → 사용자가 필드를 채우고 ✓ 체크      │
└──────────────┬──────────────────────┘
               │
               ▼
┌─────────────────────────────────────┐
│  처리 뷰                             │
│  필터: 입력완료 = true               │
│      AND 완료 = false               │
│                                     │
│  → 사용자가 항목을 작업하고 완료 표시  │
└──────────────┬──────────────────────┘
               │
               ▼
┌─────────────────────────────────────┐
│  완료 뷰                             │
│  필터: 완료 = true                   │
│                                     │
│  → 아카이브, 리포팅, 참조             │
└─────────────────────────────────────┘
```

이것은 네 가지 운영 문제를 해결합니다:

1. **명확성** — 새 항목이 어디서 시작하는지 모두가 압니다
2. **집중** — 처리 뷰에는 실행 가능한 작업만 포함됩니다
3. **이력** — 완료 뷰가 활성 작업을 어지럽히지 않고 완료된 레코드를 저장합니다
4. **자동화 준비** — 체크박스/선택 상태가 신뢰할 수 있는 전환 트리거가 됩니다

> [!NOTE]
> **설계 원칙:** 저장 위치가 아니라 필터로 단계를 분리하십시오. 세 뷰 모두 같은 데이터를 보여줍니다 — 필터가 흐름을 만듭니다.

## 상태 라이프사이클 설계

상태 필드는 장식이 아닙니다. 방향이 있는 여정입니다.

상태가 모호하면 ("Doing", "Done", "Etc") 사용자는 다음 행동이 불명확해서 망설입니다. 좋은 라이프사이클 설계는 각 상태가 행동적으로 의미 있게 만듭니다.

**설계 규칙:**
1. 상태는 임의의 라벨이 아니라 **의사결정에 관련된 단계**를 나타냅니다
2. 전환 방향이 새로운 사용자에게도 자명합니다
3. 모든 상태가 구체적인 다음 행동을 암시합니다

### 라이프사이클 예시

| 도메인 | 라이프사이클 |
|--------|----------|
| **독서** | `Wishlist → Ready → Reading → Read → Studying → Study Complete → Review Written` |
| **할 일** | `Inbox → Clarified → Scheduled → In Progress → Blocked → Done` |
| **실험** | `Idea → Designed → Running → Analyzing → Validated → Documented` |

이것들은 범용 템플릿이 아니라 아키텍처 예시입니다. 여러분의 할 일은 실제 워크플로우를 각 상태가 의도와 다음 행동을 알려주는 여정으로 모델링하는 것입니다.

> [!TIP]
> 이것을 잘 설계하면, 필터, 보드, 리포트, 리마인더가 모두 쉬워집니다 — 하나의 일관된 상태 머신을 참조하기 때문입니다.

## 페이지 구조

워크스페이스가 성장하면, 페이지 수준의 정보 아키텍처가 데이터베이스 설계만큼 중요해집니다.

### 3가지 페이지 유형

| 유형 | 목적 | 포함 내용 |
|------|---------|---------|
| **Master Table** | 데이터 창고 | 인라인 마스터 테이블 |
| **Storage** | 창고 지구 | 마스터 테이블 페이지 그룹 |
| **Information** | 전시 쇼룸 | 일상 사용을 위한 Linked Database View |

```text
워크스페이스
└── 지식 허브 (Information)
    ├── 일일 입력 (Linked View)
    ├── 독서 대시보드 (Linked View)
    ├── 주간 리뷰 (Linked View)
    └── 데이터 저장소 (Storage)
        ├── Book_Master (Master Table)
        ├── Notes_Master (Master Table)
        └── Mission_Master (Master Table)
```

> [!NOTE]
> **핵심 분리:** Information 페이지에서 **운영**하고, Storage/Master 페이지에서 **유지보수**합니다. 이 분리가 일상 사용 중 실수로 구조를 편집하는 것을 방지합니다.

## 케이스 스터디: 디지털 도서관

위의 모든 원칙을 대규모로 적용한 사례입니다. 캡처에서 실행, 분석까지의 완전한 시스템입니다.

<details>
<summary><strong>디지털 도서관 전체 아키텍처 (클릭하여 펼치기)</strong></summary>

### 핵심 아키텍처: 9개 모듈

1. **Input** — 새 항목, 목표, 미션 캡처 영역
2. **Library Master** — 정식 도서 레코드 (제목, 저자, 출판사, 페이지 수, 카테고리, 표지)
3. **Copy Collector** — 특정 도서에 연결된 주목할 만한 인용문
4. **Search** — 제목 및 인용문 키워드 빠른 검색 레이어
5. **Reports** — 월간/연간 성과 요약
6. **Information** — 카테고리/저자/출판사 인벤토리 개요
7. **Book Gallery** — 상태별 시각적 탐색
8. **Book Pause** — 상태 관리 및 전환
9. **Book Calendar** — 날짜 기반 독서 활동 추적

### 실전 상태 라이프사이클

```text
Wishlist → Ready → Reading → Read → Studying → Study Complete → Review Written
```

각 전환은 행동에 대응합니다:

| 상태 | 의미 |
|-------|---------|
| Wishlist | 후보 대기열 |
| Ready | 선택 및 준비 완료 |
| Reading | 능동적 소비 중 |
| Read | 1회 완독 |
| Studying | 추출/종합 단계 |
| Study Complete | 심층 작업 완료 |
| Review Written | 산출물 발행 완료 |

이것은 "다 읽었는데 아무것도 안 했다" 표류를 방지합니다.

### 미션 시스템 (일일 실행)

미션 레이어는 목표를 일일 행동으로 변환합니다:
- 관련 도서 선택
- 읽은 페이지 범위 기록
- 날짜/습관 맥락 첨부
- 입력완료 체크 → 레코드가 Input에서 이동

같은 기저 구조의 Linked View를 사용하기 때문에, 미션 항목은 진행 상황 및 리포트 맥락에서 즉시 사용 가능합니다.

### 상호 참조

Copy Collector는 Library 레코드와 관계를 맺습니다:
- 도서 페이지에서 → 수집된 모든 인용문 확인
- 인용문에서 → 원본 도서와 페이지로 이동
- Search는 제목 또는 인용문 텍스트를 대상으로 검색 가능

이것은 고립된 하이라이트를 재사용 가능한 지식 그래프로 변환합니다.

### 리포팅

| 지표 | 이유 |
|--------|-----|
| 목표 도서 수 vs 완료 수 | 목표 추적 |
| 읽은 페이지 수 | 볼륨 |
| 독서 일수 | 일관성 |
| 달성률 | 성취 % |
| 분포 지표 (평균, 중앙값, 분산) | 실제 행동 패턴 |

단일 퍼센트는 오해를 줄 수 있습니다. 볼륨, 일관성, 분산 신호가 필요합니다.

### 엔드투엔드 흐름

1. Input에 도서 추가 (최소 필수 필드)
2. 값 완성 → Ready로 전환
3. 읽기 시작 → 매일 미션 진행 기록
4. 완독 시 Read로 이동
5. 학습 중 Copy Collector에서 인용문 추출
6. 학습 완료 및 리뷰 작성
7. 월간 리포트 점검 → 다음 달 목표 선정 조정

</details>
